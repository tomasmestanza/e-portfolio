Codio Activity - Exploring Python tools and features

Part I
In this example, you will compile and run a program in C using the Codio workspace provided (Buffer Overflow in C). The program is already provided as bufoverflow.c - a simple program that creates a buffer and then asks you for a name, and prints it back out to the screen.

#include
int main(int argc, char **argv)
{
   char buf[8]; // buffer for eight characters
   printf("enter name:");
   gets(buf); // read from stdio (sensitive function!)
   printf("%s\n", buf); // print out data stored in buf
   return 0; // 0 as return value
}


Now compile and run the code. To test it, enter your first name (or at least the first 8 characters of it) you should get the output which is just your name repeated back to you.

Run the code a second time (from the command window this can be achieved by entering ./bufoverflow on the command line). This time, enter a string of 10 or more characters.

What happens?
An error is shown and the execution gets aborted.

What does the output message mean?
The buffer size (the memory separated or allocated for the elements of the array) is exceeded. The gets function provided by the header (stdio) doesn’t not control the input’s size.

Team Discussion: What is a Secure Programming Language?

You should read Chapter 2,6,7,8 of the course text (Pillai, 2017) and Cifuentes & Bierman (2019) and then answer the questions below, adding them as evidence to your e-portfolio.

1. What factors determine whether a programming language is secure or not?

A secure language should provide the following characteristics.

  • Allow implementation of Confidentiality, integrity and availability
  • Strong security protection
  • Resilience
  • Abstraction and provided buffer error.

2. Could Python be classed as a secure language? Justify your answer.

While we can notice security issues with different versions of Python, like serialising objects or overflow errors, making software secure is a practice more than a concept of characteristic. However, the language used for development has to provide the tools to implement security concepts.

Due to its popularity, Python is also vulnerable in terms of libraries that can create security holes, and third-party components can bring vulnerabilities If these are compromised with security breaches.

From the point of view of containing the necessary tools to prevent security issues, I consider Python a secure language. Security is an ongoing task, and a big percentage depends on the implementation.

Collaborative Discussion: Summary Post

My initial thoughts about the deserialisation of untrusted data, outlined by OWASP (N.D.), included the security risks of consuming serialised objects without prior validation.

After further research and considering an important suggestion by a fellow student, I agree that a more structured process has to be applied to avoid the consumption of untrusted serialised data. Security measures during serialisation, transmission and deserialisation are essential to mitigate the potential effects of this vulnerability.

The first of these three processes (serialisation) can only be applied if the data to be consumed is, in a way, part of the same system. Being able to participate in the serialisation process will allow the use of less complex data formats and avoid the serialisation of sensitive data (Trad, 2022).

On the transmission of the data, TLS/SSL certificates and a digital signature are advised to evaluate the data integrity (Trad, 2022).

Once the data arrives at the requester, validation and sanitation should be performed along with other security measures. At this point, having an updated list of measures is critical. These security measures have to be tested periodically.
