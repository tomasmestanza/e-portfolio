Introduction to Secure Software Development

Reading notes on Introduction to Secure Software Development

Cybersecurity is an ongoing task. It is constantly evolving and requires commitment from organisations to apply the latest measures to stop cybercrime and prevent or mitigate its effects.

Standards are used to secure software systems, and these define good practices. Independent parties develop these standards.

Integrating Security (Agile Approach)

In an iterative agile approach to managing software development, the developers can easily return to earlier stages in the life cycle without much difficulty.

From testing, for example, a developer can return to redesign the system. The iterative approach to development suits software projects which produce modular microservices and which are spread out for development across a team.

An agile approach depends on good communication, and a software product will be delivered to a customer once it is agreed that it meets their requirements. Otherwise, the system will enter another iteration of development.

As mentioned earlier, there are a variety of agile approaches to development. These include: Scram, pair programming, DSDM, Test-driven development

As agile is based on short iterations, integrating security into it can make it difficult. Prioritising the quality of software can make the identification of security concerns, not a priority, and issues can be carried out from one stage to another.

Deserialisation of untrusted data happens when the data consumer de-serialises data without proper verification (CWE, N.D.). Potential invalid data can cause security issues when an external serialised object is consumed.

Execution of an API call in the background and then sent through the caller interface as serialised data for it to be de-serialised can potentially cause security issues as it is an external object/array. Some verification action has to be applied before de-serialisation to ensure the resultant data will be valid.

For example, job systems process external APIs that provide data to it in an a-synchronised way. The interfaces involved can only send the processed data as a serialised array. The data is then sent to the requester and used by the local system without proper verification.

One of the consequences of this vulnerability is the change in the code execution through malicious objects resulting from the de-serialisation operation (OWASP, N.D.).
