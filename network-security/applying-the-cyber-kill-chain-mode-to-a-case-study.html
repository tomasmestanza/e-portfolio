Unit 2 Seminar - The Solar Winds Breach Case Study

Case notes

A routine software update that was infected and used by the malicious code as a means to attack others. Once it gets run by the victims, hackers can communicate with their computers. A connection to the Internet was essential for the attack to succeed. Through the network, hundreds of companies and federal services were affected. Hackers also got to the security and cybersecurity agency. Orion is a network monitoring software, and an update was made to it. Attackers roamed around American computer networks for nine months, and the potential of large-scale damage was the main nightmare for authorities.

As a result, other less obvious targets now might need more protection. Hackers modified sealed software code, created a system that used domain names to select targets and mimicked the Orion software communication protocols so they could hide in plain sight. And then, they did what any good operative would do: They cleaned the crime scene so thoroughly investigators can't prove definitively who was behind it. The White House has said unequivocally that Russian intelligence was behind the hack. Russia, for its part, has denied any involvement.

It started with a little piece of code modified that returned a Boolean, this was a test to se if the sealed code could e modified. So in my view, a cracking process run before anything. In the second implant of code, they added a snipped to tell them when developers build something (new feature), this was after checking in the new code by solarwinds.

During compilation, Hackers created a temporary update file containing the malicious code. So this malicious code told the system to swap the malicious temp file instead of the solar winds one. So switching fees happened last second before executable was created, this to avoid being spotter by the solar winds audit.

The malicious version of the software included code that would give the hackers unfettered, undetected access to any Orion user who downloaded and deployed the update and was connected to the Internet.

Hackers targeted gov entities and they mimicked the way Orion communicated with servers so they couldn't be identified. So once the target was identified they could come active and manipulate files, etc.

Early signs were not reported.

A second warning came after three months. A Palo Alto cybersecurity company worked with Solar Winds for three months but found nothing.

FireEye company discover finally the intrusion, this was because they had an intrusion too. After, they will discover that it was not just and intrusion but someone stealing lots of testing tools from them.

Evidence that this couldn't just have happened to any software company, Solarwinds marketing site had a list of clients, including gov agencies that used their Orion product.

This case also reminds us about password strength and multifactor authentication.

Hackers used sophisticated methods.

CKC Stage Solar Wind Case Mitigation
Reconnaissance A function that returned a boolean. (Reverse engineering). Marketing site/number of organisations as clients (installations on federal agencies). A small code was installed to assess the vulnerability. Checksum or MD5/SHA hashing to recognise changes in files
Weaponisation ‘Cracking’ sealed software (executable). Swapping update files just before finalising compilation. Use right levels of code obfuscation to prevent reverse engineering.
Delivery Routine software update. Quicker response to report (warnings) from other companies/users
Exploitation Network monitoring software. Logging & automatic report of 'keywords' in log files
Installation Executing the software update.
Command & Control ‘impersonation’, Orion’s communication to clients server.
Actions & Objectives Stealing information / used for espionage.
